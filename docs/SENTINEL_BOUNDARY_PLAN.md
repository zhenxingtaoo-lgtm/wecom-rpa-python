# 哨兵边界批量转发方案

## 背景

用户通常无法确定本次需要转发的群聊精确数量，只知道大概范围。实际操作中，会用 3 个员工私聊作为列表边界：脚本从选择聊天弹窗底部往上批量选择会话，当再次遇到这 3 个员工私聊时，说明本轮目标群已经处理到边界。

## 核心规则

- 选择方向是从下往上。
- 每批最多选择 9 个会话，符合企业微信单次发送限制。
- 3 个员工私聊是结束哨兵，不是发送目标。
- 正常批次没有哨兵时，直接发送已选会话。
- 尾批遇到哨兵时，不取消整批，而是截断本批。

## 尾批截断

如果本批已选列表中出现哨兵：

1. 按“从下往上”的选择顺序找到第一个哨兵。
2. 保留哨兵下面已经选中的群聊。
3. 移除哨兵本身以及哨兵上方所有已选项。
4. 如果保留数量大于 0，发送保留项后结束任务。
5. 如果保留数量等于 0，取消弹窗并结束任务。

换句话说：

- 哨兵下面的群聊：发送。
- 哨兵本身：不发送。
- 哨兵上面的会话：不发送。
- 发送尾批后结束任务。

示例，按从下往上的已选顺序：

```text
群A, 群B, 群C, 群D, 群E, 员工哨兵1, 员工哨兵2, 员工哨兵3
```

最终只发送 `群A..群E`，删除 3 个哨兵，然后结束。

## 自动化策略

不要每批人工确认。脚本应默认自动运行，仅在异常时停止并截图：

- 左侧已勾选候选列表识别失败。
- 哨兵识别不唯一或顺序异常。
- 已选数量和预期不一致。
- 无法删除哨兵及其上方项。
- 发现可能重复发送。

停止时如果已经进入可能影响真实发送的阶段，目标状态写为 `uncertain`，续跑时默认阻塞。人工需要根据日志、截图和企业微信聊天记录确认这批到底是否已发送，再把对应状态调整为 `sent` 或 `pending`。

## 实现方向

1. 增加哨兵配置：员工私聊名称或后续头像模板。
2. 增加左侧已勾选候选列表解析能力，得到已选会话顺序。
3. 明确内部使用“从下往上”的顺序做截断判断。
4. 增加左侧取消勾选能力，按行点击复选框移除不应发送的项。
5. 取消勾选后再次复查左侧蓝色勾选数量，确认剩余数量等于将发送数量。
6. 先用 dry-run 和截图验证截断行为，再做真实发送测试。

## 断点续跑规则

- `sent` / `skipped`：续跑时跳过。
- `pending` / `selected` / `failed`：续跑时继续处理。
- `uncertain`：续跑时停止，不自动重发。

`uncertain` 的核心含义是“程序不能证明这批没有产生发送影响”。它不是普通失败，不能由程序自动重试。
